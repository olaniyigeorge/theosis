I

Scope

Two weeks does not buy the full ontology in the PRD. It buys a graph engine generic enough that every later entity type is a data decision, not an architecture change - seeded with a thin, demo-able vertical slice.

In this build

  • StorySlot and Being nodes, with Relationships as first-class edges
  • Scripture references resolved live, never stored as text
  • Being Journey - following a person through Scripture
  • Interactive knowledge graph view + timeline strip
  • AI draft → human review → publish pipeline
  • Keyword and semantic search over nodes

Deferred, not designed-out

  • Place, Theme, Covenant, Kingdom, Prophecy - schema-ready, UI hidden
  • Multiple chronology models, scholarly-position confidence UI
  • Multi-user review, public contributions
  • Hebrew / Greek source integration
II

Architecture

Three services, each replaceable on its own timeline. The frontend never talks to Postgres or the AI providers directly - everything routes through the API.

Anthropic · Gemini · OpenAI

Scripture text is never stored in this database. Only (book, chapter, verse, translation) tuples are persisted - the text itself is resolved on read, through a cache table, so a translation license or provider swap never touches stored data.

III

Data Model

One generic nodes table with a type discriminator, one edges table for relationships, and a polymorphic scripture_refs table that both point back to. Adding Place or Covenant later will be a product decision, not a migration.

nodes

StorySlot, Being, and future entity types, discriminated by type, with type-specific fields in data JSONB.

embedding VECTOR(1536) · confidence · review_status

edges

Relationships as their own rows, not foreign keys - so they can carry confidence, notes, and review state independently.

source_id → target_id · relationship_type

scripture_refs

Polymorphic over node or edge. The enforcement mechanism for "where is this supported in Scripture?"

entity_type · book · chapter_start · verse_start

reviews

Audit trail mirroring the draft → in_review → approved → published transitions, one row per decision.

entity_id · status · reviewer · notes
IV

AI Workflow

Any provider can generate a draft. None of them can publish one.

draft→ in_review→ approved→ published

The system prompt lives in one shared file, not duplicated per provider. Every claim - node field or relationship - must cite a specific (book, chapter, verse), and every returned reference is re-validated against the Bible API before insert, regardless of which model produced it.
